# Xiaohongshu (XHS) Anti-Bot Security Analysis

Based on a deep curl and inspection of `https://www.xiaohongshu.com/explore` and its payload `xhs_explore.html`, along with the reverse engineering of `xhs_as.js` (the core anti-spider module).

## 1. The Native Polling Mechanism (Why you stayed logged in)

In the raw HTML, XHS embeds a massive JSON payload inside `<script>window.__INITIAL_STATE__ = {...}</script>`.
Inside this payload, there is a critical config:

```javascript
appSettings: {
  notificationInterval: 30, // seconds
  ...
}
```

**Finding:** When the XHS tab is left open, the website _already_ natively polls their servers every 30 seconds to check for notifications and keep the session perfectly alive. This explains why my previous hypothesis (that a 10-minute timeout killed your session) was fundamentally flawed, and why your test worked flawlessly without logging you out.

## 2. The Anti-Spider Module (`xhs_as.js`)

XHS loads a deeply obfuscated telemetry script:
`<script src="https://as.xiaohongshu.com/api/sec/v1/ds?appId=xhs-pc-web" defer="defer"></script>`

I downloaded this 5,000+ line script (`/scratch/xhs_as.js`) and ran it through a de-minifier.
It utilizes classic array-shifting obfuscation:

```javascript
var _0x341b = ["__proto__", "setPrototypeOf", "slice", "construct", "prototype", "apply", "call", "bind", "length", "toString", "keys", "isArray", "from", "push", "[object Arguments]", ...];
```

**Findings on What it Collects/Detects:**

- **Native Function Tampering:** It checks `toString` on native functions to see if they've been overwritten by bots (e.g. `function () { [native code] }`).
- **Headless Browsers:** It actively looks for `navigator.webdriver`.
- **Telemetry:** It feeds this data to generate the heavily encrypted `X-s` header used in all XHS API requests.

## 3. Vulnerabilities in Our Current Defense

1. **The `fetch` Heartbeat:** Because XHS requires the `X-s` signature generated by `xhs_as.js`, sending a bare `fetch()` to `/explore` in the background (which is what we originally implemented) is highly risky. While it might not instantly log you out, it flags the IP because it lacks the AS telemetry signatures.
2. **The Auto-Relogin Script (`relogin.py`):** We are using standard Playwright. The `xhs_as.js` script can trivially detect standard Playwright via `navigator.webdriver`. If XHS flags the browser during the QR-code login, they might silently mark the account as "high risk," causing sessions to drop much faster later on.

## 4. The Path to Absolute Victory

To achieve true, unkillable session persistence, we need to adapt our strategy based on these real findings, rather than assumptions.
